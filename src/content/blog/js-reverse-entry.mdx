很多人刚开始做 JS 逆向，习惯打开源码就开始追函数名。其实大多数时候，最先该看的不是代码本身，而是输入和输出。

因为逆向的目标通常很明确：

- 某个参数是怎么生成的
- 某段密文是怎么加密的
- 某个请求为什么带上浏览器就能过，离开浏览器就不行

## 先把边界找出来

我一般会先做三件事：

1. 在浏览器里定位哪个请求带了关键参数。
2. 记下参数原始值、请求时机和上下文。
3. 搜索这个参数名，或者搜索它附近稳定的字面量。

如果能从“请求里有一个 `sign`”收缩到“项目里只有 3 个地方会碰它”，后面的工作量会直接降下来。

## 别一上来就怕混淆

混淆确实烦，但它经常只是视觉噪音，不一定是最难的部分。

真正影响逆向效率的，通常是这些问题：

- 不知道哪个函数才是真正入口
- 不知道参数依赖哪些上游变量
- 不知道执行时机
- 不知道有没有环境检测

所以我更喜欢先打日志、下断点、改返回值，而不是急着全量反混淆。

## 一个够用的思路

当我怀疑某个函数参与了签名生成时，最常做的是：

- 看它的入参是什么
- 看它返回什么
- 看它有没有调用 Web API、时间函数或随机数
- 看它是否依赖 `window`、`document`、`navigator`

如果一个函数的输入已经足够像原始参数，输出又刚好像目标签名，那它大概率就在主链路上。

```js
function trace(fn, name) {
  return function (...args) {
    console.log(name, "args:", args);
    const result = fn.apply(this, args);
    console.log(name, "result:", result);
    return result;
  };
}
```

## 逆向不是证明自己看得懂全部代码

我越来越觉得，逆向最重要的能力不是“把所有代码都读懂”，而是“只读必须读懂的那一段”。

如果目标只是还原一个参数生成流程，那就没必要把整套业务逻辑都理清。能稳定复现，就是阶段性胜利。
