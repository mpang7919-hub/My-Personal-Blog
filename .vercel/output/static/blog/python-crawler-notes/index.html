从一个请求开始：我怎么拆一条 Python 爬虫链路

写爬虫时，最怕的不是代码多，而是没有顺序。先把链路拆开，问题就会开始变简单。

2026年4月24日 2 分钟阅读
#Python 爬虫 #实战 #网络请求

很多爬虫问题，表面上看是“拿不到数据”，但真正的问题往往不是代码不会写，而是链路没有拆开。

我现在做一个新站点时，通常不会急着上框架，也不会一开始就写并发。第一步永远是把一条请求走通，然后确认下面几件事：

先确认数据到底从哪里来

最常见的误区，是在页面源码里翻很久，却没有先确认目标数据到底来自哪里。

我一般会先看这几类入口：

  • 页面初始化时内嵌的数据
  • XHR 或 fetch 请求
  • 翻页或懒加载触发的接口
  • 某个加密参数拼出来之后才会返回的数据

如果数据根本不在 HTML 里，那继续盯着解析 BeautifulSoup 往往是在浪费时间。

再确认请求能不能脱离浏览器复现

把请求从浏览器里抄出来，只是开始，不是结束。

我会继续确认：

  • 请求头里哪些字段是真的必要
  • Cookie 是不是强依赖
  • 参数里有没有时间戳、签名、随机数
  • 响应是否和环境绑定，比如 UA、Referer、TLS 指纹

在这个阶段，最有价值的不是“把它抄成 Python 代码”，而是知道哪些字段可以删，哪些字段一删就死。

import requests

url = "https://example.com/api/list"
params = {"page": 1}
headers = {"User-Agent": "Mozilla/5.0"}

resp = requests.get(url, params=params, headers=headers, timeout=10)
print(resp.status_code)
print(resp.text[:200])

最后才是工程化

只有当单条链路足够稳定时，我才会继续做这些事情：

  • 抽分页逻辑
  • 加重试和限速
  • 处理代理
  • 落库或导出
  • 做调度

顺序不能反。链路都没看明白，就开始封装类、抽配置、写线程池，最后只会让问题更难排查。

我的一个小原则

如果某个爬虫问题两小时还没进展，我会强迫自己回到最原始的问题：

“我要的数据，到底是从哪一个请求、在什么条件下返回的？”

很多时候，答案其实早就在 Network 面板里，只是人容易一开始就写太多代码。
