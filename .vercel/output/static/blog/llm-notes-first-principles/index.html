刚开始折腾 AI 大模型时，我更关心上下文，而不是参数表

模型效果不好时，很多问题不在温度或 top_p，而在任务描述、上下文材料和输出约束。

2026年4月22日 2 分钟阅读
#AI 大模型 #Prompt #实践

刚开始接触大模型时，我也会不停翻参数文档，想知道 temperature、top_p、max_tokens 该怎么调。

后来做得多了，反而越来越觉得，影响结果质量最大的，通常不是这些参数，而是你给了模型什么上下文。

先想清楚模型到底要完成什么

很多“大模型不稳定”的问题，本质上是任务定义不稳定。

比如你让模型“分析这段代码”，这句话其实非常模糊。它到底要：

  • 找 bug
  • 解释逻辑
  • 重构代码
  • 补测试
  • 发现安全问题

如果目标不清楚，模型只能自己猜。而只要它开始猜，输出就很容易飘。

比调参数更重要的是材料组织

我现在更在意的是这些问题：

  • 给模型的输入材料是否完整
  • 有没有混入无关上下文
  • 输出格式是否定义清楚
  • 成功标准是否可验证

比如把需求、限制条件、输入样例、期望输出都明确写出来，往往比单独调 temperature 更有效。

一个很朴素但有用的方法

我经常把任务改写成这种结构：

任务目标：
输入材料：
限制条件：
输出格式：
判定标准：

这样做的好处是，模型更容易沿着你给的轨道走，而不是凭空补全一个它以为你想要的答案。

大模型更像协作者，不像魔法按钮

我现在不太期待模型“一次就给完美答案”。更现实的方式，是把它当成一个能快速迭代的协作者：

  • 先给它足够清楚的任务边界
  • 再让它产出第一版结果
  • 根据偏差继续修正上下文

如果上下文足够清楚，模型通常会比你想象得更稳定。反过来，如果上下文是乱的，再高级的模型也只能在混乱里尽力发挥。
